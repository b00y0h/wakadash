# Phase 7: Distribution - Research

**Researched:** 2026-02-19
**Domain:** Homebrew distribution (personal tap + homebrew-core submission)
**Confidence:** HIGH

## Summary

Phase 7 implements Homebrew distribution through a two-track approach: immediate distribution via a personal tap (b00y0h/homebrew-wakadash) using GoReleaser's `homebrew_casks` feature, and longer-term submission to homebrew-core for broader discoverability.

The personal tap provides instant user access via `brew tap b00y0h/wakadash && brew install wakadash`. It uses casks (not formulae) because wakadash distributes pre-built binaries via GitHub releases. GoReleaser v2.10+ introduced `homebrew_casks` specifically for this use case, replacing the deprecated `brews` section. The critical requirement is a Personal Access Token with write access to the tap repository, since GitHub's default `GITHUB_TOKEN` cannot write across repositories.

The homebrew-core submission is a separate, more involved process. It requires building from source (not using pre-built binaries), meeting strict acceptance criteria (>=30 forks, >=30 watchers, or >=75 stars for self-submitted projects), and passing comprehensive audits. wakadash currently has 0 stars/forks/watchers (created 2026-02-19), so homebrew-core acceptance will require community adoption first. However, the formula can be prepared and submission attempted — the worst case is delayed acceptance until popularity thresholds are met.

A critical 2026 change: Homebrew 5.0.0 deprecated `--no-quarantine` and will remove unsigned/unnotarized casks from official taps by September 2026. Personal taps can still use quarantine removal hooks, but this is explicitly a security tradeoff that must be documented transparently. Code signing requires a $99/year Apple Developer Program membership.

**Primary recommendation:** Use GoReleaser `homebrew_casks` for the personal tap with quarantine removal hooks (documented security tradeoff). Create a separate homebrew-core formula that builds from source. Submit the formula PR immediately but expect delayed acceptance until community metrics improve.

## Architecture Patterns

### Pattern 1: Personal Tap with GoReleaser Casks

**What:** GoReleaser automatically generates and publishes Homebrew cask to tap repository on each GitHub release.

**When to use:** Every release for CLI tools distributed as pre-built binaries.

**Configuration:**
```yaml
# .goreleaser.yaml
version: 2

# ... existing builds, archives, checksum sections ...

homebrew_casks:
  - name: wakadash

    # Tap repository configuration
    repository:
      owner: b00y0h
      name: homebrew-wakadash
      token: "{{ .Env.HOMEBREW_TAP_TOKEN }}"

    # Cask metadata
    homepage: https://github.com/b00y0h/wakadash
    description: "Live terminal dashboard for WakaTime coding stats"
    license: MIT

    # Cask file location (default: Casks)
    directory: Casks

    # Git commit configuration
    commit_author:
      name: goreleaserbot
      email: bot@goreleaser.com
    commit_msg_template: "Brew formula update for {{ .ProjectName }} version {{ .Tag }}"

    # macOS quarantine removal for unsigned binaries
    hooks:
      post:
        install: |
          if OS.mac?
            system_command "/usr/bin/xattr", args: ["-dr", "com.apple.quarantine", "#{staged_path}/wakadash"]
          end

    # Skip pre-release versions
    skip_upload: auto
```

**Key requirements:**
- Repository must be named `homebrew-wakadash` (the `homebrew-` prefix enables short form `brew tap b00y0h/wakadash`)
- `HOMEBREW_TAP_TOKEN` must be a Personal Access Token with `contents: write` permission on the tap repository
- The default `GITHUB_TOKEN` cannot write to separate repositories
- Quarantine removal hook is necessary for unsigned binaries but bypasses macOS Gatekeeper

**Source:** [GoReleaser Homebrew Casks Documentation](https://goreleaser.com/customization/homebrew_casks/), [GoReleaser v2.10 Announcement](https://goreleaser.com/blog/goreleaser-v2.10/)

### Pattern 2: Homebrew-Core Formula (Build from Source)

**What:** A manually-created formula that builds wakadash from source using Go's build system.

**When to use:** For homebrew-core submission where pre-built binaries are not acceptable.

**Formula structure:**
```ruby
# Formula/w/wakadash.rb
class Wakadash < Formula
  desc "Live terminal dashboard for WakaTime coding stats"
  homepage "https://github.com/b00y0h/wakadash"
  url "https://github.com/b00y0h/wakadash/archive/refs/tags/v0.2.0.tar.gz"
  sha256 "PLACEHOLDER_SHA256"
  license "MIT"

  depends_on "go" => :build

  def install
    # Set ldflags for version injection (matches GoReleaser pattern)
    ldflags = %W[
      -s -w
      -X main.version=#{version}
      -X main.commit=#{tap.user}
      -X main.date=#{time.iso8601}
    ]

    system "go", "build", *std_go_args(ldflags: ldflags), "./cmd/wakadash"

    # Generate shell completions if supported
    generate_completions_from_executable(bin/"wakadash", "completion")
  end

  test do
    # Test that binary runs and outputs version
    assert_match version.to_s, shell_output("#{bin}/wakadash --version")

    # Test that config loading fails gracefully without ~/.wakatime.cfg
    # (Don't test actual dashboard since that requires WakaTime API key)
    output = shell_output("#{bin}/wakadash 2>&1", 1)
    assert_match "wakatime.cfg", output
  end
end
```

**Key differences from personal tap cask:**
- Builds from source using `go build` (not installing pre-built binary)
- Uses `std_go_args` helper for standard Go build flags
- Includes `test do` block with meaningful validation (not just `--version`)
- No quarantine removal hook (homebrew-core requires proper code signing by September 2026)
- SHA256 checksum of source tarball (not binary)

**Source:** [Homebrew Formula Cookbook](https://docs.brew.sh/Formula-Cookbook), [gh formula example](https://github.com/Homebrew/homebrew-core/blob/master/Formula/g/gh.rb), [hugo formula example](https://github.com/Homebrew/homebrew-core/blob/master/Formula/h/hugo.rb)

### Pattern 4: Personal Access Token Setup

**What:** Create a fine-grained PAT with write access to the tap repository.

**When to use:** Required for GoReleaser to publish casks to the tap repository.

**Setup steps:**
1. Go to GitHub Settings → Developer settings → Personal access tokens → Fine-grained tokens
2. Click "Generate new token"
3. Configure:
   - **Token name:** `wakadash-homebrew-tap`
   - **Expiration:** 1 year (set calendar reminder to renew)
   - **Repository access:** Only select repositories → `homebrew-wakadash`
   - **Permissions:** Contents (Read and write)
4. Generate token and copy the value
5. Add to wakadash repository secrets:
   ```bash
   gh secret set HOMEBREW_TAP_TOKEN -b"<token-value>" -R b00y0h/wakadash
   ```
6. Update `.github/workflows/release.yml` to pass the token:
   ```yaml
   env:
     GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
     HOMEBREW_TAP_TOKEN: ${{ secrets.HOMEBREW_TAP_TOKEN }}
   ```

**Why fine-grained PAT:** Classic PATs grant access to all repositories owned by the user. Fine-grained PATs scope to specific repositories with specific permissions, reducing security risk.

**Source:** [GoReleaser GitHub Actions Documentation](https://goreleaser.com/ci/actions/), [Creating Your First Homebrew Tap](https://kristoffer.dev/blog/guide-to-creating-your-first-homebrew-tap/)

### Anti-Patterns to Avoid

- **Using `brews` instead of `homebrew_casks`:** The `brews` section is deprecated as of GoReleaser v2.10 (June 2025). Use `homebrew_casks` for pre-built binaries.
- **Using GITHUB_TOKEN for tap publishing:** The default action token cannot write to other repositories. Always use a separate PAT.
- **Omitting quarantine removal hook in personal tap:** Unsigned binaries will trigger macOS Gatekeeper warnings without this hook.
- **Using binary URL in homebrew-core formula:** homebrew-core requires building from source. Use the tarball URL, not release binary URLs.
- **Weak test block:** Testing only `--version` or `--help` is discouraged. Test actual functionality.
- **Submitting fork to homebrew-core:** Forks are rejected unless officially designated as successors.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Cask generation for tap | Custom Ruby scripts | GoReleaser `homebrew_casks` | Handles checksums, versioning, binary paths, git commits automatically |
| Formula creation for homebrew-core | Copy-paste from other formulas | `brew create <url>` then manual editing | `brew create` generates boilerplate with correct structure |
| Cross-repository authentication | Hardcoded tokens or multiple PATs | Fine-grained PAT with minimal scope | Fine-grained PATs limit blast radius if compromised |
| Local testing of cask | Installing from GitHub releases manually | `brew install --cask /path/to/tap/Casks/wakadash.rb` | Tests actual cask installation flow |

**Key insight:** Homebrew distribution has complex edge cases (platform-specific binaries, SHA256 verification, formula syntax, audit requirements). GoReleaser solves the tap automation problem completely. For homebrew-core, the manual formula approach is required but well-documented.

## Common Pitfalls

### Pitfall 1: Token Permission Errors

**What goes wrong:** Release succeeds but cask isn't published to tap. GitHub Actions logs show "Resource not accessible by integration" or similar permission errors.

**Why it happens:** Using `secrets.GITHUB_TOKEN` instead of a Personal Access Token with write access to the tap repository. The default action token only has permissions for the repository running the workflow.

**How to avoid:**
1. Create a fine-grained PAT with `contents: write` permission on the tap repository
2. Add it as repository secret: `HOMEBREW_TAP_TOKEN`
3. Reference in `.goreleaser.yaml`: `token: "{{ .Env.HOMEBREW_TAP_TOKEN }}"`
4. Pass in GitHub Actions workflow: `HOMEBREW_TAP_TOKEN: ${{ secrets.HOMEBREW_TAP_TOKEN }}`

**Warning signs:** Release artifacts appear on GitHub, but no commit appears in tap repository. Check Actions logs for "permission denied" or "authentication" errors.

**Source:** [GoReleaser GitHub Actions documentation](https://goreleaser.com/ci/actions/), Phase 3 research

### Pitfall 2: "Application is Damaged" on macOS

**What goes wrong:** Users install via personal tap, but macOS prevents execution with "wakadash is damaged and can't be opened. You should move it to the Trash."

**Why it happens:** macOS applies quarantine attribute to downloaded binaries. Without code signing, Gatekeeper blocks unsigned binaries. This became stricter with macOS 13+ (Ventura) and Homebrew 5.0.0.

**How to avoid:** Add post-install hook to cask:
```yaml
hooks:
  post:
    install: |
      if OS.mac?
        system_command "/usr/bin/xattr", args: ["-dr", "com.apple.quarantine", "#{staged_path}/wakadash"]
      end
```

**Warning signs:** GitHub issues reporting "damaged application" immediately after fresh install on macOS 13+.

**Important note:** This bypasses macOS security. Document transparently in tap README. Alternative is paying $99/year for Apple Developer Program and implementing code signing + notarization.

**Source:** [Homebrew 5.0.0 announcement](https://workbrew.com/blog/homebrew-5-0-0), [Removing support for `--no-quarantine`](https://github.com/Homebrew/brew/issues/20755)

### Pitfall 3: Tap Repository Naming

**What goes wrong:** Creating repository as `wakadash-homebrew` or `tap-wakadash`. Users cannot use short form `brew tap b00y0h/wakadash`.

**Why it happens:** Homebrew requires `homebrew-` prefix for short form tap command to work.

**How to avoid:** Always name tap repositories `homebrew-<name>`. For wakadash, the repository must be `homebrew-wakadash`. This enables:
- Short form: `brew tap b00y0h/wakadash`
- Instead of: `brew tap b00y0h/wakadash https://github.com/b00y0h/homebrew-wakadash`

**Warning signs:** Documentation shows full GitHub URL in tap command.

**Source:** [Homebrew Taps documentation](https://docs.brew.sh/Taps)

### Pitfall 4: Homebrew-Core Popularity Threshold

**What goes wrong:** Formula PR is closed or stalled due to insufficient popularity metrics.

**Why it happens:** Homebrew-core has explicit requirements for self-submitted software: ">=30 forks, >=30 watchers or >=75 stars." wakadash currently has 0/0/0 (repository created 2026-02-19).

**How to avoid:**
- Submit formula PR early but acknowledge the metrics gap transparently
- Continue development and promotion to build community
- Keep PR updated as metrics improve
- Alternative: Wait until thresholds are met before submitting

**Warning signs:** PR comments asking about project popularity or community adoption.

**Expected timeline:** homebrew-core acceptance may take months until community adoption grows. Personal tap provides distribution in the meantime.

**Source:** [Acceptable Formulae](https://docs.brew.sh/Acceptable-Formulae)

### Pitfall 5: GoReleaser-Generated Formula in homebrew-core

**What goes wrong:** Attempting to submit the formula generated by GoReleaser's `brews` section to homebrew-core. PR is rejected or requires extensive rework.

**Why it happens:** GoReleaser formulas install pre-built binaries. homebrew-core requires building from source with `go build`.

**How to avoid:** Create a separate, manually-written formula for homebrew-core using the pattern in Pattern 2. The GoReleaser-generated cask and the homebrew-core formula serve different purposes:
- **Cask (personal tap):** Installs pre-built binary from GitHub release
- **Formula (homebrew-core):** Builds from source tarball

**Warning signs:** Formula contains download URLs for binary artifacts instead of source tarball. Formula uses `bin.install` instead of `system "go", "build"`.

**Source:** [GoReleaser docs note](https://goreleaser.com/customization/homebrew_formulas/): "goreleaser also does not generate valid homebrew-core formulas at the moment"

### Pitfall 6: Missing Test Block Validation

**What goes wrong:** Formula test block only checks `--version` or `--help`. Audit fails or maintainers request better tests.

**Why it happens:** Version/help flags are easy but don't validate actual functionality.

**How to avoid:** Test meaningful behavior. For wakadash:
```ruby
test do
  # Test version output
  assert_match version.to_s, shell_output("#{bin}/wakadash --version")

  # Test that config loading fails gracefully (without API key)
  output = shell_output("#{bin}/wakadash 2>&1", 1)
  assert_match "wakatime.cfg", output
end
```

**Warning signs:** `brew audit` suggests improving test block. PR reviewers request better validation.

**Source:** [Formula Cookbook](https://docs.brew.sh/Formula-Cookbook) emphasizes "bad tests" that only check version

### Pitfall 7: Unsigned Casks in Official Taps After September 2026

**What goes wrong:** Personal tap works fine in 2026, but in September 2026 Homebrew starts removing unsigned casks from official taps.

**Why it happens:** Homebrew 5.0.0 deprecated unsigned casks and set September 2026 as the enforcement deadline.

**How to avoid:** Understand the tradeoffs:
- **Personal tap:** Can continue using unsigned binaries with quarantine removal hooks indefinitely
- **homebrew-core:** Will require code signing by September 2026
- **Options:**
  1. Stay with personal tap (no code signing required)
  2. Invest in Apple Developer Program ($99/year) for code signing
  3. Accept that homebrew-core may not be viable without code signing

**Timeline impact:** Phase 7 can complete successfully with unsigned binaries in personal tap. homebrew-core acceptance may be blocked by signing requirements in late 2026.

**Source:** [Homebrew 5.0.0 announcement](https://workbrew.com/blog/homebrew-5-0-0)

## Open Questions

1. **When to submit homebrew-core formula?**
   - What we know: wakadash has 0 stars/forks/watchers. Threshold is >=30 forks, >=30 watchers, or >=75 stars for self-submitted projects.
   - What's unclear: Whether early submission with transparent acknowledgment is preferred, or whether to wait until thresholds are met.
   - Recommendation: Submit early with clear note in PR description that metrics are below threshold but formula is ready for review. This allows formula improvements while community grows. Worst case: PR stays open until popularity improves.

2. **Code signing investment timing?**
   - What we know: Apple Developer Program costs $99/year. Unsigned binaries work in personal taps but will be removed from official taps by September 2026.
   - What's unclear: Whether wakadash will gain enough popularity to justify code signing investment.
   - Recommendation: Defer code signing until (a) homebrew-core acceptance looks likely, or (b) September 2026 approaches and project has active users. Personal tap with quarantine removal hook is sufficient for Phase 7.

3. **Shell completions support?**
   - What we know: wakadash uses stdlib `flag` package. The homebrew-core formula template includes `generate_completions_from_executable` but this typically requires Cobra or similar framework.
   - What's unclear: Whether shell completions provide enough value to justify adding Cobra dependency.
   - Recommendation: Defer shell completions to a future phase. Not required for initial homebrew-core submission. Can be added later if community requests it.

4. **Tap vs homebrew-core marketing?**
   - What we know: Personal tap provides immediate distribution. homebrew-core provides discoverability (appears on brew.sh, in `brew search`).
   - What's unclear: Whether to actively promote the personal tap or focus on building community for homebrew-core acceptance.
   - Recommendation: Promote personal tap immediately in README and release notes. Mention homebrew-core as "planned" but set expectations that acceptance requires community adoption first.
