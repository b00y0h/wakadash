# Phase 3: Homebrew Distribution - Research

**Researched:** 2026-02-13
**Domain:** Homebrew tap distribution with GoReleaser
**Confidence:** HIGH

## Summary

Homebrew distribution for Go CLI tools has evolved significantly. As of June 2025, GoReleaser v2.10 introduced the `homebrew_casks` section, replacing the deprecated `brews` section. This change aligns with Homebrew's philosophy: formulae build from source, while casks install pre-built binaries. Since wakafetch is a Go binary distributed via GitHub releases, it should use casks, not formulae.

The implementation requires creating a separate tap repository (b00y0h/homebrew-wakafetch), configuring GoReleaser to publish casks on release, and managing authentication tokens for cross-repository publishing. Homebrew's 2026 code signing requirements mean unsigned binaries need post-install hooks to remove quarantine attributes.

**Primary recommendation:** Use GoReleaser's `homebrew_casks` configuration to automatically publish to a dedicated tap repository on each release. Include post-install quarantine removal hook for macOS compatibility.

### Pattern 1: Automated Cask Publishing on Release
**What:** GoReleaser automatically generates and commits Homebrew cask to tap repository when GitHub release is created.

**When to use:** Every release workflow for CLI tools distributed as binaries.

**Example workflow:**
```yaml
# .github/workflows/release.yml
name: Release
on:
  push:
    tags:
      - 'v*'
jobs:
  goreleaser:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with:
          go-version: stable
      - uses: goreleaser/goreleaser-action@v6
        with:
          version: '~> v2'
          args: release --clean
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          HOMEBREW_TAP_TOKEN: ${{ secrets.HOMEBREW_TAP_TOKEN }}
```

**Key insight:** GITHUB_TOKEN cannot write to separate repositories. A separate Personal Access Token (HOMEBREW_TAP_TOKEN) with `contents: write` permission on the tap repository is required.

### Pattern 3: Quarantine Removal for Unsigned Binaries
**What:** Post-install hook removes macOS quarantine attribute so binary runs without "damaged" warning.

**When to use:** When not paying for Apple Developer Program code signing ($99/year).

**Example:**
```yaml
hooks:
  post:
    install: |
      if OS.mac?
        system_command "/usr/bin/xattr", args: ["-dr", "com.apple.quarantine", "#{staged_path}/wakafetch"]
      end
```

**Security consideration:** This bypasses macOS Gatekeeper. Document this in README so users understand the security tradeoff.

**Source:** [GoReleaser Homebrew Casks documentation](https://goreleaser.com/customization/homebrew_casks/)

### Anti-Patterns to Avoid
- **Using `brews` instead of `homebrew_casks`:** The `brews` section is deprecated as of GoReleaser v2.10 (June 2025) and will be removed in v3. Formulae are for building from source; casks are for pre-built binaries.
- **Using default GITHUB_TOKEN for tap publishing:** GitHub's default action token cannot write to other repositories. This will fail silently or error during release.
- **Hardcoding binary name in quarantine removal:** Use `#{staged_path}/wakafetch` not a hardcoded path. The staged path varies.
- **Omitting tap repository README:** Users expect documentation. An empty repository looks abandoned and unprofessional.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Homebrew cask generation | Custom Ruby scripts to generate .rb files | GoReleaser `homebrew_casks` | GoReleaser handles version updates, SHA256 checksums, binary installation, and git commits automatically. Edge cases include multiple architectures, version templating, and proper Ruby syntax. |
| Binary SHA256 checksums | Manual checksum calculation and updating | GoReleaser automatic checksums | GoReleaser computes and embeds checksums from build artifacts. Manual checksums are error-prone and break automation. |
| Cross-repository publishing | Custom git automation scripts | GoReleaser repository configuration + PAT | GoReleaser handles git clone, file generation, commit, and push. Custom scripts miss edge cases like merge conflicts, branch detection, and commit message formatting. |
| macOS quarantine handling | Custom installation scripts or user instructions | GoReleaser post-install hooks | Post-install hooks run automatically during `brew install`. Custom scripts require users to remember extra steps, reducing adoption. |

**Key insight:** Homebrew distribution has complex edge cases (architecture-specific binaries, checksum verification, version updates, commit signing). GoReleaser has solved these problems across thousands of projects. Custom automation will miss edge cases and require ongoing maintenance.

## Common Pitfalls

### Pitfall 1: Token Permission Errors
**What goes wrong:** Release succeeds but cask isn't published to tap. Logs show "Resource not accessible by integration" or permission errors.

**Why it happens:** Using `secrets.GITHUB_TOKEN` instead of a Personal Access Token (PAT) with write access to the tap repository. GitHub's default action token only has permissions for the repository running the action.

**How to avoid:**
1. Create a Personal Access Token with `contents: write` permission
2. Add it as repository secret (e.g., `HOMEBREW_TAP_TOKEN`)
3. Reference in GoReleaser config: `token: "{{ .Env.HOMEBREW_TAP_TOKEN }}"`
4. Pass in GitHub Actions: `HOMEBREW_TAP_TOKEN: ${{ secrets.HOMEBREW_TAP_TOKEN }}`

**Warning signs:** Release completes successfully, but no commit appears in tap repository. Check GitHub Actions logs for permission errors.

**Source:** [GoReleaser GitHub Actions documentation](https://goreleaser.com/ci/actions/), [goreleaser-action issue #58](https://github.com/goreleaser/goreleaser-action/issues/58)

### Pitfall 2: "Application is Damaged" macOS Warning
**What goes wrong:** Users install via Homebrew, but macOS prevents execution with "wakafetch is damaged and can't be opened. You should move it to the Trash."

**Why it happens:** macOS applies quarantine attribute to downloaded binaries. Without Apple code signing, Gatekeeper blocks execution. This became stricter in recent macOS versions and Homebrew 5.0.0 (November 2025).

**How to avoid:** Add post-install hook to remove quarantine attribute:
```yaml
hooks:
  post:
    install: |
      if OS.mac?
        system_command "/usr/bin/xattr", args: ["-dr", "com.apple.quarantine", "#{staged_path}/wakafetch"]
      end
```

**Warning signs:** GitHub issue reports about "damaged application" on fresh installs. macOS version 13+ (Ventura and later) are most affected.

**Source:** [GoReleaser Homebrew Casks documentation](https://goreleaser.com/customization/homebrew_casks/), [Homebrew cask issue #51538](https://github.com/Homebrew/homebrew-cask/issues/51538)

### Pitfall 3: Wrong Homebrew Artifact Type
**What goes wrong:** Using `brews` (deprecated) instead of `homebrew_casks` in .goreleaser.yml. This still works in GoReleaser v2 but will break in v3.

**Why it happens:** Older documentation and examples use `brews`. The change to `homebrew_casks` happened in June 2025 (v2.10), so tutorials written before then are outdated.

**How to avoid:** Use `homebrew_casks` for binary distribution. Only use `brews` (now deprecated) if you're actually building from source, which is not the case for Go binaries released via GoReleaser.

**Warning signs:** GoReleaser deprecation warnings in release logs. Documentation references "formulae" instead of "casks" for pre-built binaries.

**Source:** [GoReleaser v2.10 announcement](https://goreleaser.com/blog/goreleaser-v2.10/), [GoReleaser deprecation notices](https://goreleaser.com/deprecations/)

### Pitfall 4: Tap Repository Naming Convention
**What goes wrong:** Creating tap repository as `wakafetch-homebrew` or `tap-wakafetch` instead of `homebrew-wakafetch`. Users can't use short form `brew tap b00y0h/wakafetch`.

**Why it happens:** Homebrew requires `homebrew-` prefix for the short form tap command to work. Without it, users must use the full GitHub URL.

**How to avoid:** Always name tap repositories `homebrew-{name}`. GoReleaser will automatically use the short form `{owner}/{name}` in generated casks.

**Warning signs:** Users have to run `brew tap b00y0h/wakafetch https://github.com/b00y0h/wakafetch` instead of just `brew tap b00y0h/wakafetch`.

**Source:** [Homebrew Taps documentation](https://docs.brew.sh/Taps), [How to Create and Maintain a Tap](https://docs.brew.sh/How-to-Create-and-Maintain-a-Tap)

### Pitfall 5: Missing Cask Verification Test
**What goes wrong:** Cask publishes successfully but binary path is wrong or binary doesn't work. Users discover this only after installation.

**Why it happens:** Unlike formulae, casks don't have `test do` blocks. Testing requires manual verification or CI automation. Binary paths can change between versions or architectures.

**How to avoid:**
1. Test locally after each GoReleaser change: `brew install --cask b00y0h/wakafetch/wakafetch`
2. Run `wakafetch --version` to verify binary works
3. Check binary is in PATH: `which wakafetch`
4. Add verification to release checklist

**Warning signs:** User issues reporting "command not found" after Homebrew installation. Binary installed but not in PATH.

**Note:** Homebrew casks don't support `test do` blocks like formulae. Validation must be external.

**Source:** [Homebrew cask issue #18830](https://github.com/Homebrew/homebrew-cask/issues/18830), [Cask Cookbook](https://docs.brew.sh/Cask-Cookbook)

### Pitfall 6: Tap Repository Initialization
**What goes wrong:** Pushing to tap repository fails because it doesn't exist or has no commits. GoReleaser can't clone an empty repository.

**Why it happens:** Created GitHub repository but forgot to initialize with README or first commit. GoReleaser needs a valid git history to clone and push to.

**How to avoid:**
1. Create repository with README: `gh repo create homebrew-wakafetch --public --description "Homebrew tap for wakafetch"`
2. Initialize if created empty: Add README.md and commit before first GoReleaser run
3. Verify repository is accessible and has at least one commit

**Warning signs:** GoReleaser logs show "repository not found" or "couldn't clone" errors during release.

**Source:** General git/GitHub behavior, [gh CLI documentation](https://cli.github.com/manual/gh_repo_create)

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `brews` section in .goreleaser.yml | `homebrew_casks` section | June 2025 (GoReleaser v2.10) | Formulae are for building from source; casks are proper approach for pre-built binaries. Using brews was a hack that's now deprecated. |
| Manually managing Homebrew formulae in tap repositories | GoReleaser automation with git commits | Ongoing since ~2017 | Automation eliminates manual checksum calculation, Ruby syntax errors, and forgotten version updates. |
| Submitting to homebrew-core for all tools | Personal taps for niche/new tools | Ongoing | homebrew-core has strict requirements (popularity, stability, maintenance). Personal taps allow immediate distribution without approval process. |
| No code signing requirements | Homebrew 5.0.0 deprecates unsigned casks | November 2025, enforcement September 2026 | Unsigned casks will be removed from official taps. Personal taps can still use quarantine removal hooks, but users need to understand security tradeoffs. |

**Deprecated/outdated:**
- **`brews` section:** Deprecated in GoReleaser v2.10 (June 2025), will be removed in v3. Use `homebrew_casks` instead.
- **Formula approach for binaries:** Homebrew formulae are for building from source. Pre-built Go binaries should use casks.
- **Using GITHUB_TOKEN for cross-repository publishing:** GitHub's default token cannot write to other repositories. Requires separate PAT.
- **Unsigned binaries without quarantine removal:** As of macOS 13+ and Homebrew 5.0.0, stricter Gatekeeper enforcement means unsigned binaries need explicit quarantine removal or will be blocked.

## Open Questions

1. **Code Signing vs Quarantine Removal**
   - What we know: Apple Developer Program costs $99/year for code signing. Quarantine removal hook bypasses this but requires user trust.
   - What's unclear: User perception and adoption impact. Do users avoid installing unsigned binaries from personal taps?
   - Recommendation: Start with quarantine removal hook. Document security tradeoff transparently in README. Revisit code signing if wakafetch gains significant adoption.

2. **Tap Repository Visibility**
   - What we know: Tap repository should be public for Homebrew to access. Cask file is auto-generated.
   - What's unclear: Should tap repository have additional documentation, contribution guidelines, or issue templates?
   - Recommendation: Keep tap repository minimal (README only) initially. Main repository is source of truth for issues and documentation. Add to tap only if users commonly open issues there.

3. **Cask Naming Conflicts**
   - What we know: Personal taps avoid most naming conflicts. If submitting to homebrew-core eventually, may need unique name.
   - What's unclear: Whether "wakafetch" conflicts with existing casks (low probability but unchecked).
   - Recommendation: Use "wakafetch" as-is. Personal tap namespace (b00y0h/wakafetch) provides uniqueness. Only revisit if submitting to homebrew-core in future.
